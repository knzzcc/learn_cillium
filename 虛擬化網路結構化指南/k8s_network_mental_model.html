🤔 Pod 在不同 Node 上，封包怎麼送過去？ ▶
Pod IP 是叢集虛擬的，實體網路不認識。所以需要某種方式把 Pod 封包「藏」在 Node 之間的真實封包裡送過去。這就是隧道的概念。 IP-in-IP VXLAN
生活比喻：你寄一封信（Pod 封包），但郵局不認識收件地址。所以你把這封信塞進一個大信封（外層標頭），大信封寫的是郵局認識的地址（Node IP）。到了目的地再拆開大信封，取出原本的信。
IP-in-IP只加 20 bytes 外層 IP。輕薄快速，但只能搬 L3 封包。Calico 預設用這個。
VXLAN多加 ~50 bytes，把整個乙太網路幀包進 UDP。更通用，什麼網路環境都能跑。Flannel 預設用這個。
🧪 動手試
在 Node 上跑 ip -d link show tunl0（IP-in-IP）或 ip -d link show flannel.1（VXLAN），你會看到隧道介面。再用 tcpdump -i eth0 -n proto 4 抓 IP-in-IP 封包，親眼看到外層標頭。
🤔 每個 Node 怎麼知道「某個 Pod IP 在哪個 Node 上」？ ▶
隧道解決了怎麼「送」，但送之前 Node 得知道該送去「哪」。這就是路由資訊的分發問題。 BGP
生活比喻：想像每個 Node 是一個社區的管理員。BGP 就是管理員之間的 Line 群組：「10.244.1.0/24 那些住戶在我這邊」「10.244.2.0/24 在我這邊」。大家互相廣播，每個人都知道該把信送去哪個社區。

Calico 用 BGP 讓每個 Node 把自己負責的 Pod 子網廣播出去。如果所有 Node 在同一個 L2 子網，甚至可以純用 BGP 路由，完全不需要隧道封裝（ipPool.ipipMode: Never），效能最好。跨子網時才回退到 IP-in-IP。

🧪 動手試
裝了 Calico 後跑 calicoctl node status，你會看到 BGP peer 列表和連線狀態。再跑 ip route，會看到類似 10.244.2.0/24 via 192.168.1.20 dev tunl0 的路由——這就是 BGP 學來的。
🤔 iptables 規則太多太慢，有沒有更高效的方式處理封包？ ▶
當叢集有上千個 Service，kube-proxy 產生的 iptables 規則可能破萬條。每個封包都要線性比對，延遲愈來愈高。 eBPF
生活比喻：iptables 像是在門口設了一排警衛，每個人進來都要從第一個警衛問到最後一個。eBPF 像是在門口裝了一台人臉辨識機——直接查表，O(1) 時間搞定，不用排隊。而且這台機器跑在 Linux 核心裡面，封包根本不用繞到使用者空間。

Cilium 這個 CNI 就是用 eBPF 取代 iptables 和 kube-proxy。它直接在核心層攔截封包、做負載平衡和 NetworkPolicy 過濾，不需要產生任何 iptables 規則。

🧪 動手試
裝 Cilium 後跑 cilium status 看 eBPF 狀態，再用 cilium bpf lb list 查看 Service 的 eBPF 負載平衡表。對比 iptables -t nat -L | wc -l 看規則數——你會發現幾乎是空的。

串起來的記憶方式

先問問題，再記答案：

❶ 封包怎麼跨 Node？→ 隧道（IP-in-IP 輕薄 / VXLAN 通用）
❷ Node 怎麼知道路由？→ BGP 互相廣播
❸ 規則太多太慢？→ eBPF 在核心層查表

每個技術都是一個問題的解法，不是一個要背的名詞。在自己的叢集上跑一次上面的指令，比讀十篇文章都有效。
