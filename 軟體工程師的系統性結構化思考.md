## Claude responded: 系統性結構化思考（Systematic & Structured Thinking）

### 系統性結構化思考（Systematic & Structured Thinking）

這是軟體工程師在面對複雜問題時，能夠**有條理、有層次地分析與解決問題**的能力。具體來說包含幾個核心面向：

#### 1. 分解問題（Decomposition）

把大問題拆成小問題，再各個擊破。例如設計一個電商系統，不是想「整個系統怎麼做」，而是拆成：用戶系統、商品系統、訂單系統、支付系統……

#### 2. 抽象化（Abstraction）

忽略不重要的細節，專注在核心邏輯。例如設計 API 時先定義介面（interface），不急著想實作細節。

#### 3. 模式識別（Pattern Recognition）

看到新問題時，能識別出它跟哪些已知問題類似，套用已有的解法（設計模式、演算法）。

#### 4. 由上而下 / 由下而上思考

- **Top-down**：先定架構，再填細節
- **Bottom-up**：先確認基礎元件可行，再往上組合

#### 5. 考量邊界條件與例外

不只想「正常情況」，也系統性地思考：空值、極端值、並發、失敗情境……

#### 6. 權衡取捨（Trade-off Analysis）

結構化地比較方案，例如用表格列出各方案在效能、維護性、開發成本上的差異。

------

**簡單說：** 就是遇到問題不是憑感覺亂試，而是能**有框架地拆解 → 分析 → 決策 → 驗證**，讓思考過程可重複、可溝通、可審查。

這也是為什麼面試常考系統設計題——考的不只是答案，而是你**思考的過程夠不夠結構化**。

## Claude responded: 八股文系統化準備框架

### 八股文系統化準備框架

#### 第一步：建立知識地圖（先不要背）

把八股文題目分類成「知識樹」：

```
後端八股
├── 作業系統
│   ├── 進程 vs 執行緒
│   ├── 記憶體管理
│   └── 死鎖
├── 網路
│   ├── TCP/UDP
│   ├── HTTP/HTTPS
│   └── 三次握手
├── 資料庫
│   ├── 索引原理
│   ├── 事務 ACID
│   └── SQL vs NoSQL
└── 資料結構與演算法
    ├── 複雜度分析
    └── 常見結構
```

先把**地圖畫出來**，知道自己要學什麼、缺什麼。

------

#### 第二步：每個題目用同一個模板回答

避免答案散亂，強迫自己套框架：

```
【是什麼】一句話定義
【為什麼存在】解決什麼問題
【怎麼運作】核心原理
【優缺點 / 適用場景】
【跟相似概念的差異】例如 TCP vs UDP
【實際應用例子】
```

這樣回答面試時有頭有尾，不會答到一半沒話說。

------

#### 第三步：主動回憶，不要反覆閱讀

| 錯誤做法          | 正確做法                      |
| ----------------- | ----------------------------- |
| 看筆記 → 覺得懂了 | 蓋住筆記 → 自己講出來         |
| 反覆讀同一篇文章  | 間隔重複（今天學、3天後複習） |
| 死背定義          | 用自己的話解釋給虛構的人聽    |

#### 第四步：建立連結，不要孤立記憶

把知識點**串起來**，例如：

> TCP 三次握手 → 為什麼不是兩次？→ 因為要確認雙向都能收發 → 這跟 HTTPS 的 TLS handshake 有什麼關係？

**問自己「為什麼」和「跟什麼有關」**，比死背有效十倍。

------

#### 第五步：輸出驗證

| 方式           | 說明                         |
| -------------- | ---------------------------- |
| 口頭自問自答   | 模擬面試，計時回答           |
| 寫成自己的筆記 | 不 copy 別人，強迫消化       |
| 教別人         | 講不清楚的地方就是還沒真正懂 |

#### 實際執行節奏建議

```
每天 1~2 個主題
→ 畫出該主題的子知識點
→ 用模板整理答案（手寫或打字）
→ 隔天先回憶，再對照補漏
→ 一週後做一次整體串聯複習
```

------

**核心原則：** 八股文的敵人不是「題目太多」，是**沒有結構導致記了又忘**。用框架學，學一個紮一個。
