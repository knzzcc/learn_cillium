場景一：資料庫
最常見
MySQL、PostgreSQL、MongoDBRWO
資料庫需要穩定的低延遲讀寫，一次只有一個節點存取磁碟就夠了。
雲端首選
AWS EBS (gp3/io2)、GCP Persistent Disk SSD
雲端原生 block storage，裝好 CSI driver 就能用。gp3 是通用型，io2 可以指定更高的 IOPS 給重負載 DB。
自建首選
Rook-Ceph RBD 或 Longhorn
Ceph RBD 提供分散式 block storage，資料自動複製到多個節點防止硬碟壞掉遺失。Longhorn 更簡單，適合不想花太多時間在儲存維運的團隊。
場景二：多個 Pod 共享檔案
需要 RWX
CMS 上傳目錄、ML 訓練資料集、共用設定檔RWX
多個 Pod 同時讀寫同一份檔案，block storage 做不到，必須用檔案系統層級的方案。
雲端首選
AWS EFS、GCP Filestore
託管式 NFS，不用自己架設。EFS 按用量計費彈性高，Filestore 效能更穩定但要先選容量。
自建首選
CephFS 或 NFS Server
CephFS 是 Ceph 的檔案系統模式，適合大規模。小規模用一台 NFS server 就夠了，但它是單點故障。
場景三：大量非結構化資料
不用 PV
圖片、影片、Log 檔、備份檔不走 PV/PVC
幾 TB 甚至幾十 TB 的檔案不適合放進 PV，管理困難又貴。改用物件儲存，讓應用直接透過 SDK 存取。
雲端
S3、GCS、Azure Blob
便宜、無限擴展、不需要掛載。應用程式用 SDK（例如 AWS SDK）直接讀寫。
自建
MinIO
S3 相容的開源物件儲存，可以跑在 K8s 裡。API 跟 S3 幾乎一樣，程式碼不用大改。
場景四：機密與憑證
特殊用法
API Key、TLS 證書、資料庫密碼Ephemeral volume
不走傳統 PV/PVC。CSI driver 把機密以檔案形式注入 Pod，Pod 消失機密就消失。
方案
Secrets Store CSI Driver + Vault / 雲端 KMS
比 K8s 原生 Secret 安全，機密不存在 etcd 裡，而是從 Vault 或雲端 KMS 即時拉取。
場景五：暫存與快取
最簡單
CI/CD build cache、暫存檔、中間運算結果不需要 CSI
資料丟了也沒關係，只是為了加速。
方案
emptyDir 或 hostPath
emptyDir 隨 Pod 生滅，最簡單。hostPath 可以存取節點的本地磁碟，但 Pod 被調度到別的節點就讀不到了。
快速判斷法： 先看存取模式——只要一個 Pod 寫就選 block (RWO)，多個 Pod 共寫就選檔案系統 (RWX)，量大到 TB 級就跳過 PV 用物件儲存。
