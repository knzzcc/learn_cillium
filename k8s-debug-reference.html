⚙️ CRI 架構總覽

🔗 什麼是 CRI？

Container Runtime Interface (CRI) 是 Kubernetes 定義的 gRPC 介面標準，讓 kubelet 可以透過統一的介面與不同的容器執行環境溝通，而不被綁定在特定實作上。

CRI 主要定義兩組服務：RuntimeService（管理 Pod sandbox 與容器的生命週期）和 ImageService（管理映像檔的拉取、列表、刪除）。

📦 各層角色說明

kubelet：節點代理，從 API Server 接收 PodSpec，透過 CRI 呼叫 runtime 建立/銷毀容器。

containerd / CRI-O：高階 runtime，負責映像檔管理、容器 metadata、呼叫低階 runtime。containerd 是目前最主流的選擇（Docker Engine 底層也是它）。CRI-O 是 Red Hat 主導、專為 K8s 打造的輕量替代。

runc / crun：低階（OCI）runtime，實際執行 clone()、設定 namespace、cgroup、seccomp profile，然後 exec 容器的 entrypoint。runc 是 Go 寫的參考實作，crun 是 C 寫的，啟動速度更快。

kata-runtime：用輕量級 VM 取代 namespace 隔離，提供更強的安全邊界。

🔍 CRI 層級的 Debug
# 用 crictl 直接跟 CRI runtime 溝通（繞過 kubectl） crictl ps # 列出容器 crictl pods # 列出 pod sandbox crictl inspect <container-id> # 容器詳細資訊 crictl logs <container-id> # 容器日誌 crictl images # 列出映像檔 crictl stats # 容器資源用量 # ctr 是 containerd 原生 CLI ctr -n k8s.io containers list ctr -n k8s.io tasks list # 檢查 containerd / CRI-O 服務狀態 systemctl status containerd journalctl -u containerd --since "10 min ago"
何時用 crictl？ 當 kubectl 無法正常運作（例如 API Server 掛了、kubelet 異常）時，直接到節點上用 crictl 是唯一的手段。也常用於確認「容器是否真的在跑」以排除 kubelet 層的問題。
🐋 Docker 在這個架構中的位置

Kubernetes 1.24 起移除了 dockershim。現在的 Docker Engine 底下其實也是 containerd，所以如果節點裝的是 Docker，K8s 會直接跟 containerd 溝通，Docker CLI 只是另一個 containerd 的前端。

如果是純 Docker 環境（非 K8s），debug 指令就是 docker inspect、docker logs、docker exec 等。

🧱 Linux Namespace 隔離

📐 容器 = Namespace + Cgroup + Root FS

容器本質上不是虛擬機，而是利用 Linux kernel 的三個機制實現隔離：

Namespace 隔離「看得到什麼」—— 每個容器有自己的 PID 樹、網路堆疊、掛載點等。Cgroup 限制「用得了多少」—— CPU、記憶體、I/O 的用量上限。OverlayFS / Root FS 提供獨立的檔案系統視角。

🗂️ 七種 Namespace
Namespace隔離什麼Debug 意義
PIDProcess ID 樹容器內看不到宿主機 process；shareProcessNamespace 可打破
NET網路堆疊（介面、路由、iptables）每個 Pod 一個 net ns；同 Pod 容器共享；netshoot 掛入同 ns debug
MNT掛載點容器的檔案系統視角；hostPath volume 會穿越隔離
UTSHostname容器 hostname 預設 = Pod name
IPC共享記憶體、semaphore同 Pod 容器共享 IPC ns
USERUID/GID 映射user namespace 啟用後 root in container ≠ root on host
CGROUPCgroup 根目錄視角影響 /proc/self/cgroup 看到的路徑
🔬 從宿主機檢視 Namespace
# 找到容器的 PID（在宿主機上） crictl inspect <container-id> | jq '.info.pid' # 查看該 PID 的 namespace ls -la /proc/<pid>/ns/ # 進入容器的 network namespace nsenter -t <pid> -n ip addr nsenter -t <pid> -n ss -tlnp # 進入所有 namespace（等同 docker exec） nsenter -t <pid> -m -u -i -n -p /bin/sh
nsenter 是 debug 神器。當容器內沒有 shell 或工具時（distroless image），從宿主機用 nsenter 進入特定 namespace，搭配宿主機上的工具來除錯，比如進 net ns 跑 tcpdump。
⚡ Cgroup 檢查
# cgroup v2（現代系統）查看記憶體限制 cat /sys/fs/cgroup/<cgroup-path>/memory.max cat /sys/fs/cgroup/<cgroup-path>/memory.current # 查看 CPU 配額 cat /sys/fs/cgroup/<cgroup-path>/cpu.max # 容器內簡易查看 cat /proc/self/cgroup cat /sys/fs/cgroup/memory.max # cgroup v2

🔐 權限模型

🛡️ 權限的三道防線

容器的權限由三個層級共同決定，由外而內依序是：

1. Linux Capabilities — 將傳統的 root 全權拆成細粒度的能力（約 40 種），容器可以只授予需要的 cap。

2. Seccomp Profile — 白名單制過濾 system call，沒列的 syscall 直接拒絕。Docker/containerd 預設有一組 default profile 封鎖約 40+ 種危險 syscall。

3. AppArmor / SELinux — MAC（強制存取控制），限制 process 能存取哪些檔案路徑和網路操作。

🧩 常用 Linux Capabilities 一覽
Capability允許什麼誰需要
NET_RAW原始 socket（ping、tcpdump）網路 debug 工具
NET_ADMIN修改路由、iptables、介面CNI plugin、iptables debug
SYS_PTRACEptrace（strace、gdb）效能追蹤、debug attach
SYS_ADMINmount、bpf、namespace 操作eBPF 工具、特權操作
SYSLOG讀取 kernel ring bufferdmesg
DAC_OVERRIDE繞過檔案權限檢查需存取他人檔案
CHOWN改變檔案擁有者init container 修復權限
NET_BIND_SERVICE綁定 <1024 port容器內跑 80/443
Docker 預設授予約 14 個 cap（包括 CHOWN、NET_RAW 等），K8s 預設跟隨 runtime。生產環境最佳實務是在 PodSpec 中 drop: ["ALL"] 再 add 需要的。
📝 K8s SecurityContext 設定
# Pod spec 範例 securityContext: runAsNonRoot: true runAsUser: 1000 fsGroup: 2000 seccompProfile: type: RuntimeDefault containers: - name: app securityContext: allowPrivilegeEscalation: false capabilities: drop: ["ALL"] add: ["NET_BIND_SERVICE"] readOnlyRootFilesystem: true

🌐 網路 Debug 工具

🔌 ss — Socket 狀態速查
ss -tlnp # TCP listening, 顯示 port + process ss -tnp # 所有 TCP 連線（established 等） ss -s # socket 統計摘要 ss -ti # TCP 內部資訊（cwnd, rtt, retrans） ss -tnp state time-wait | wc -l # TIME-WAIT 數量 ss -tnp dst :3306 # 連往 MySQL 的連線 ss -tnp src :8080 # 從 8080 出去的連線

比 netstat 快得多，因為直接讀 kernel netlink socket 而非解析 /proc。排查 port 衝突、連線洩漏、TIME-WAIT 堆積的首選。

🛤️ ip — 網路設定檢查
ip addr # 介面 + IP 位址 ip route # 路由表（default gw 是否正確） ip neigh # ARP 表 ip link show # 介面 up/down 狀態、MTU ip -s link # 介面流量統計 + 錯誤計數 # 在宿主機上檢視所有 network namespace ip netns list ip netns exec <ns> ip addr
🦈 tcpdump / Wireshark / ksniff
# 容器內或 nsenter 後抓封包 tcpdump -i eth0 -nn port 80 tcpdump -i any -w /tmp/cap.pcap # 存檔用 Wireshark 開 # ksniff：kubectl plugin，一鍵抓 pod 封包 → Wireshark kubectl sniff <pod-name> -n <namespace> # netshoot 掛入目標容器的 net ns docker run --rm -it --net=container:<target> nicolaka/netshoot kubectl debug -it <pod> --image=nicolaka/netshoot --target=<container>
🔎 DNS / 連線測試
# DNS 解析測試 nslookup kubernetes.default.svc.cluster.local dig +short myservice.mynamespace.svc.cluster.local # 檢查 CoreDNS 是否正常 kubectl -n kube-system logs -l k8s-app=kube-dns # HTTP 連線測試 curl -v http://service-name:port/health # TCP 連通性 nc -zv db-host 5432 # MTU / 封包大小問題排查 ping -M do -s 1400 <target>
📊 nstat / /proc/net 計數器
# 查看 TCP 重傳、UDP overflow 等計數器 nstat -sz | grep -i retrans nstat -sz | grep -i drop # 或直接讀 /proc cat /proc/net/snmp # TCP/UDP/IP 層統計 cat /proc/net/netstat # 擴展 TCP 統計

大量 TCPRetransSegs 暗示網路品質差或 MTU 問題；ListenOverflows 暗示 backlog 太小。

🐝 Cilium + Hubble（eBPF 可觀測）

如果叢集使用 Cilium CNI，Hubble 提供免 sidecar 的 L3/L4/L7 流量觀測：

hubble observe --pod mynamespace/mypod hubble observe --verdict DROPPED # 只看被丟的封包 hubble observe --protocol DNS # DNS 查詢 cilium monitor --type drop # 即時看 policy drop

🖥️ 系統層 Debug 工具

📋 dmesg — Kernel 訊息
# OOM Killer 記錄 dmesg | grep -i oom dmesg | grep -i "killed process" # 網路/硬體錯誤 dmesg | grep -iE "error|fail|drop|link" # 磁碟 I/O 異常 dmesg | grep -iE "xfs|ext4|io error|blk" # 即時追蹤（需 SYSLOG cap） dmesg -w # 在 K8s 節點上（透過 debug pod） kubectl debug node/<node> -it --image=ubuntu -- dmesg | grep oom
容器內預設無法執行 dmesg（dmesg_restrict=1）。實務上在宿主機或 kubectl debug node/ 上執行。最常用於排查 OOM kill — 確認是哪個 cgroup 的哪個 process 被殺。
🔬 strace — Syscall 追蹤
# 追蹤某 process 的所有 syscall strace -p <pid> -f # 只看網路相關 syscall strace -p <pid> -e trace=network # 只看檔案操作 strace -p <pid> -e trace=open,read,write,stat # 統計各 syscall 花費的時間 strace -p <pid> -c # 在 K8s 中用 ephemeral container # Pod 需啟用 shareProcessNamespace: true kubectl debug -it <pod> --image=ubuntu --target=<container>

容器啟動後卡住？strace 通常能立刻看到卡在哪個 syscall（DNS lookup、connect、read config 等）。

📂 lsof — File Descriptor 檢查
lsof -i :8080 # 誰佔了 port 8080 lsof -p <pid> # 該 process 開的所有 fd lsof -i -P -n # 所有網路連線 # 直接讀 /proc（不需 lsof） ls /proc/<pid>/fd/ | wc -l # fd 數量 cat /proc/<pid>/limits # ulimit 設定

📡 可觀測性工具

📝 日誌收集

EFK / ELK Stack — Elasticsearch + Fluentd (或 Filebeat) + Kibana。全文索引、強大查詢，但資源消耗高。

Loki + Grafana — 只索引 label 不做全文索引，輕量許多。用 LogQL 查詢，適合 K8s label 體系。

stern — 命令列工具，同時 tail 多個 pod 的日誌並以顏色區分。stern "api-.*" -n production

kubectl logs 技巧：--previous（crash 前的日誌）、-c init-container（指定容器）、--since=5m（最近 5 分鐘）。

📈 指標監控

Prometheus + Grafana — 指標收集 + 視覺化的標配。搭配 Alertmanager 做告警。kube-state-metrics 暴露 K8s 物件狀態，node-exporter 暴露節點指標。

cAdvisor — 內建於 kubelet，提供每個容器的 CPU/MEM/NET/FS 指標。

kubectl top — 需要 metrics-server。kubectl top pod --sort-by=memory 快速找到記憶體大戶。

🔗 分散式追蹤

Jaeger / Zipkin + OpenTelemetry — 追蹤請求在微服務之間的流經路徑和各段延遲。排查「為什麼這個 API 慢」的利器。

搭配 Istio/Linkerd service mesh 可以自動注入 trace header，不需改 code 就能得到基礎追蹤。深入追蹤仍需在 application 層用 OTel SDK 埋 span。

🛡️ 安全監控

Falco — runtime 安全偵測，基於 eBPF/kernel module，偵測容器內執行 shell、讀取 /etc/shadow、異常網路連線等。

Trivy — 映像檔 CVE 掃描 + K8s manifest 設定掃描。trivy image nginx:latest

kube-bench — 檢查叢集是否符合 CIS Kubernetes Benchmark。

🖥️ 互動式管理工具

k9s — 終端機互動式 K8s dashboard。快速切換 ns、查看 pod、即時 log、exec shell。k9s -n production

Lens / OpenLens — 圖形化 K8s IDE，內建 Prometheus 整合，適合偏好 GUI 的人。

📖 Debug Playbook

💀 場景一：Pod CrashLoopBackOff
STEP 1

kubectl describe pod → 看 Events、Exit Code、Reason。Exit 137 = OOM kill 或 SIGKILL；Exit 1 = 應用程式錯誤。

STEP 2

kubectl logs --previous → 看 crash 前的日誌。加 -c <container> 指定 init 或 sidecar。

STEP 3

如果是 Exit 137 且日誌沒有應用程式錯誤：到宿主機 dmesg | grep oom 確認是否被 OOM Killer 殺掉。如果是，調高 memory limit 或排查記憶體洩漏。

STEP 4

如果是應用程式啟動失敗：kubectl debug -it <pod> --copy-to=debug-pod --container=app -- sh 建立副本 pod 進入除錯。或改 command 為 sleep infinity 進去手動啟動觀察。

🌐 場景二：Service 連不到
STEP 1

kubectl get endpoints <svc> → endpoint 是否為空？空 = selector 沒 match 或 pod 沒 ready。

STEP 2

DNS 解析：在 pod 內 nslookup <svc>.<ns>.svc.cluster.local。失敗 = CoreDNS 問題。

STEP 3

直接 curl Pod IP + port → 通 = kube-proxy/iptables 問題；不通 = 應用程式或 NetworkPolicy 問題。

STEP 4

檢查 NetworkPolicy：kubectl get networkpolicy -n <ns>，看 ingress/egress 規則是否阻擋。hubble observe --verdict DROPPED 如果有 Cilium。

STEP 5

深入抓包：kubectl sniff <pod> 或 nsenter + tcpdump，看封包到底到了哪一層。

🐌 場景三：應用程式變慢
STEP 1

kubectl top pod → CPU 是否被 throttle？cat /sys/fs/cgroup/cpu.stat 看 nr_throttled。Memory 是否接近 limit？

STEP 2

ss -ti → 看 TCP 連線的 rtt 和 retrans。高重傳 = 網路品質問題或 MTU 問題。

STEP 3

strace -c -p <pid> → 統計 syscall 時間分佈。如果 futex 佔大量時間 = 鎖競爭；epoll_wait = 等待 I/O。

STEP 4

Profiling：kubectl flame <pod> 產生火焰圖找 CPU hotspot。看 Jaeger trace 找跨服務延遲瓶頸。

💾 場景四：Volume 掛載失敗
STEP 1

kubectl describe pvc → Events 通常會說原因。常見：StorageClass 不存在、quota 滿、node affinity 不符。

STEP 2

kubectl describe pod → 看 Volume 相關 Events，特別是 FailedMount、FailedAttachVolume。

STEP 3

到宿主機：dmesg | grep -i xfs\|ext4 檢查磁碟錯誤。df -h 檢查節點磁碟空間。journalctl -u kubelet 看 mount 錯誤。

🧭 通用 Debug 心法
由外而內：kubectl describe/logs/events → 進容器 ss/curl/nslookup → 宿主機 dmesg/journalctl/nsenter → 抓包 tcpdump → 追蹤 strace/perf
不要一開始就 privileged。先用不需權限的工具（ss、ip、curl、/proc）排查，必要時才用 ephemeral container 或 debug node pod 取得更高權限。
生產環境慎用 strace/tcpdump。這些工具會對目標 process 產生效能影響。tcpdump 在高流量下可能丟包或佔用大量磁碟。建議限時限範圍。
